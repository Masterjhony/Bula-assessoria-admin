<objective>
Adicionar um checkbox de consentimento obrigatório ("Autorizo a Bula Assessoria a entrar em contato comigo no WhatsApp") ao Step 3 do formulário, sem alterar nenhum outro campo, step, handler ou lógica de submit já existente.

Purpose: O negócio precisa de uma autorização explícita de contato via WhatsApp antes de aceitar o lead. Como `submitForm` já espalha `...data` no payload, a forma mais segura de transportar esse dado é fazê-lo nascer como mais um campo de `FormData` — sem tocar na função de submit em si.

Output: `src/components/Form.tsx` com: (1) um novo tipo `FormErrors` dedicado ao mapa de mensagens de erro (necessário porque `FormData` deixa de ter só campos `string` com a chegada do booleano), (2) o campo `whatsappConsent: boolean` em `FormData` + valor inicial `false`, (3) a checagem obrigatória em `validateStep` (Step 3), (4) um handler dedicado `handleWhatsappConsentChange`, e (5) o checkbox renderizado no Step 3, logo antes dos botões Voltar/Enviar.
</objective>

  <action>
    Seis mudanças, todas dentro de `src/components/Form.tsx`, nenhuma outra:

    1. TIPO DE ERROS — logo após a linha `type FieldKey = keyof FormData`, adicionar um novo tipo: `type FormErrors = { [K in keyof FormData]?: string }`. Motivo: hoje `Partial<FormData>` é reaproveitado como o tipo do mapa de mensagens de erro, o que só funciona porque todo campo de `FormData` é `string`. Ao adicionar um campo `boolean`, essa premissa quebra — `FormErrors` mapeia toda chave de `FormData` (independente do tipo real do campo) para uma mensagem `string` opcional.

    2. CAMPO NO FORMDATA — na interface `FormData`, adicionar `whatsappConsent: boolean` imediatamente após `inscricaoEstadual: string`.

    3. TROCAR `Partial<FormData>` POR `FormErrors` nos três lugares que hoje o usam como tipo do mapa de erros: a assinatura de retorno de `validateStep` (`function validateStep(step: number, data: FormData): Partial<FormData>` → `: FormErrors`), a declaração interna `const errors: Partial<FormData> = {}` dentro de `validateStep` (→ `const errors: FormErrors = {}`), e o `useState` de erros no componente `Form` (`useState<Partial<FormData>>({})` → `useState<FormErrors>({})`). Não alterar o cast `stepErrors as Record<string, boolean>` usado nas chamadas de `trackFormValidationFailed` — ele continua funcionando sem mudança.

    4. VALIDAÇÃO — dentro do bloco `if (step === 3) { ... }` de `validateStep`, depois da linha que valida `inscricaoEstadual`, adicionar: `if (!data.whatsappConsent) errors.whatsappConsent = 'Você precisa autorizar o contato via WhatsApp para continuar.'`.

    5. ESTADO INICIAL + HANDLER — no `useState<FormData>` do componente `Form`, adicionar `whatsappConsent: false,` ao grupo de campos do Step 3 (junto de `momento: '', cabecas: '', interesse: '', quantidade: '', inscricaoEstadual: '',`). Logo abaixo de `handleUFChange` (mesmo estilo, um handler dedicado — não reaproveitar o `handleChange` genérico de string), adicionar: `function handleWhatsappConsentChange(checked: boolean) { setFormData(prev => ({ ...prev, whatsappConsent: checked })); trackFormFieldChanged('whatsappConsent', step, checked); if (errors.whatsappConsent) setErrors(prev => ({ ...prev, whatsappConsent: undefined })) }`. `trackFormFieldChanged` já aceita `(field: string, step: number, hasValue: boolean)` — `'whatsappConsent'` como string literal é válido sem nenhuma mudança na assinatura importada de `../analytics/posthog`.

    6. CHECKBOX NO STEP 3 — dentro do JSX de `{step === 3 && (...)}`, logo depois do bloco condicional `{formData.interesse && ( <div>...quantidade...</div> )}` e ainda dentro da mesma `<div className="space-y-4">` (como último filho dela, antes do `</div>` que a fecha), inserir um novo `<div>` contendo: um `<label className="flex items-start gap-3 cursor-pointer select-none">` envolvendo um `<input type="checkbox" checked={formData.whatsappConsent} onChange={e => handleWhatsappConsentChange(e.target.checked)} className="mt-0.5 h-4 w-4 shrink-0 rounded border-white/25 bg-white/5 accent-gold focus:ring-1 focus:ring-gold/50 focus:ring-offset-0 cursor-pointer" />` seguido de um `<span className="text-white/70 text-sm leading-snug">` com o texto literal exato `Autorizo a Bula Assessoria a entrar em contato comigo no WhatsApp`; logo abaixo do `</label>`, dentro do mesmo `<div>` wrapper, `{errors.whatsappConsent && <span className={errorClass}>{errors.whatsappConsent}</span>}` (mesmo padrão de erro usado pelos outros campos). A classe `accent-gold` usa o token de cor `gold` já definido em `tailwind.config.js` (`#C8A96E`), alinhando o checkbox marcado à paleta oficial da marca.

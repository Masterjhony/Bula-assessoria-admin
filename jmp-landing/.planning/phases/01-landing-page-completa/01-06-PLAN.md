<objective>
Fix mobile (and desktop) scroll-to-form behaviour so every CTA button lands the user
at the form card, not the top of the page or a random section position.

Purpose: Currently, several Footer anchors have plain href="#inscricao" with no
preventDefault, causing iOS Safari to execute a hash navigation jump rather than a
smooth scroll. Additionally, scrollIntoView targets the outer <section id="inscricao">
which sits at y=0 — so the scroll appears to "go to top". The fix adds a second,
inner scroll anchor on the form card, updates all CTAs to target it, removes the
double-smooth conflict, and ensures Footer bare anchors gain the same JS handler.

Output:
- A new id="inscricao-form" anchor placed at the top of the right-hand form column
- All 7 scroll CTAs across Header, LeilaoCard, and Footer targeting that inner anchor
- html { scroll-behavior: smooth } removed to eliminate iOS Safari double-smooth conflict
- scroll-padding-top set to account for any future sticky header (80px mobile / 96px desktop)
- Footer bare anchors (lines 17-25, 67-77) upgraded with onClick handlers
</objective>

  <name>Task 1: Add inner scroll anchor in Form and fix index.css scroll conflicts</name>
  <files>src/components/Form.tsx, src/index.css</files>
  <action>
In src/components/Form.tsx, add a new div with id="inscricao-form" as the very first
child of the right-hand form column (the div with className starting "lg:w-[52%]").
This element should be visually invisible — use className="absolute" with no dimensions,
no content, positioned so that scrollIntoView on it lands the viewport at the top of
the form card. Since the section's left column is tall on mobile, the outer section
anchor (id="inscricao") scrolls to section-top which shows the hero copy, not the
inputs. The inner anchor placed at the top of the right-side form column is what the
user actually wants to reach.

Example placement (prose description, not code):
- The right column div currently opens with: <div className="lg:w-[52%] bg-black/65 ...">
- Insert as its first child: <div id="inscricao-form" />
- No styling needed beyond the id; scroll-padding-top (set in CSS) handles the offset.

Do NOT remove id="inscricao" from the outer section — it is used for URL fragment
purposes and by the existing goTo() smooth scroll for step changes inside the form.

In src/index.css:
- Remove the line "scroll-behavior: smooth" from the html rule. Keeping it creates a
  double-smooth conflict on iOS Safari: the browser executes CSS smooth scroll for the
  hash change, then the JS scrollIntoView fires a second smooth scroll, and on Safari
  the two cancel each other out, causing a snap to the hash target (the outer section,
  which is at y=0) instead of the inner anchor.
- Add scroll-padding-top to html: 80px for all viewports, overridden to 96px at
  sm (640px+). This pre-compensates for any sticky header height so the anchor does
  not get obscured. The Header is currently not rendered in App.tsx but the scroll-
  padding-top is cheap insurance and has no visual side-effect when no header is present.

After this task, index.css html rule should read:
  html {
    scroll-padding-top: 80px;
  }
  @media (min-width: 640px) {
    html {
      scroll-padding-top: 96px;
    }
  }

The .hero-bg class and its media query remain unchanged.
